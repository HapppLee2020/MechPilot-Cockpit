1. 项目背景与目标

1.1 现状痛点

  • 硬件组 40-50 名工程师日常使用 SolidWorks + PDM，零件检入后需手动填写大量自定义属性（物料编码、名称、分类、材料、重量、表面处理等），重复劳动多，遗漏和填写不规范频发
  • 属性检查、图纸审核等质量环节依赖人工逐项核对，效率低且标准执行不一致
  • 已有的编码规则引擎（PDM MCP Server + SQL Server）能自动生成物料编码，但仅限于编码生成，未覆盖完整的属性填写流程
  • 多台工作站（安装 SW + PDM + MCP Server）已有基础设施，但尚未形成统一的任务调度体系

1.2 项目目标

目标说明验收标准
属性自动填写 工程师检入 PDM 后，一键触发 Agent 自动读取零件信息并按公司规则填写全部属性 常规零件属性填写准确率 ≥ 95%，单件处理时间 ≤ 30s
属性合规检查 检查已填写属性的完整性和规范性，输出检查报告 必填字段遗漏检出率 100%
多工作站并发 多台 SW 工作站可同时处理不同任务，Agent Server 统一调度 支持 ≥ 5 台工作站并发，单任务排队等待 ≤ 60s（非极端场景）
多用户服务 支持 40-50 名工程师同时使用，统一排队和结果推送 系统支持 ≥ 40 并发用户，任务不丢失、不串号
规则可维护 公司标准和填写规则以自然语言维护，工程师可直接编辑，改完即时生效 规则变更零代码修改，修改后下一个任务立即生效
结果通知 任务完成后通过 Add-in 面板或飞书消息推送结果报告 双通道可达，从任务完成到工程师收到通知 ≤ 10s

组件现状本平台复用方式

2.1 架构概览

2.2 核心设计原则

原则说明
工程师主动触发 任务由工程师通过 Add-in 按钮主动提交，不监听 PDM 检入事件。避免草稿/中间版本检入产生无效任务
Agent 编排，Worker 执行 Agent（LLM）负责理解任务意图、生成执行计划；Worker 负责调用 SW/PDM API 完成实际操作。两者解耦
规则即文档，文档即规则 公司标准、填写规则以 Markdown 自然语言维护，工程师直接编辑，零代码修改
最小 LLM 调用 常规任务通过固定流程 + 规则引擎处理，仅在需要推理判断时调用 LLM。降低延迟和 API 成本
执行计划缓存 LLM 生成的执行计划按文件特征缓存，相似零件直接复用历史计划，减少 LLM 调用
失败通知工程师 PDM 文件锁冲突等不可恢复错误，重试后仍失败则通知工程师处理，不无限重试

2.3 组件职责划分

组件职责技术
SW Add-in 模型数据采集、任务提交、结果展示、进度轮询 C# / .NET Framework
Agent Server API 网关、任务队列、路由分发、规则加载、Worker 管理、报告生成、消息推送 Python / FastAPI
MCP Worker 执行 SW/PDM 操作、心跳上报、执行结果回传 Python / SW-Remote-MCP (已有)
PDM Vault 文件存储与版本管理、编码规则数据源 SQL Server (已有)
LLM Cloud 任务编排推理、报告内容生成 火山方舟 / Claude API
飞书 任务完成通知推送 lark-cli (已有)

3.2 部署约束

节点网络环境约束部署要求
工程师 Desktop LAN only，无外网 无法直接访问 LLM API 安装 SW + PDM Client + Add-in DLL，配置 Server LAN IP
Agent Server Dual NIC：WAN + LAN LAN 侧固定 IP 笔记本需有线接入 LAN，防止 Wi-Fi IP 变动导致客户端断连
MCP Worker LAN only 需安装 SW + PDM，保持常驻运行 IP 相对固定，SW 实例空闲时才能接受任务
关键风险 如果 Agent Server 部署在笔记本上，笔记本休眠/关机 = 全组服务中断。建议评估是否部署到企业服务器，或至少确保笔记本长期在线且配置 Wake-on-LAN。

5.2 各状态说明

状态触发条件耗时预期Add-in 展示
QUEUED 任务提交后进入队列，等待空闲 Worker 取决于队列长度，通常 0-60s "排队中 #3"
DISPATCHED 已分配 Worker，准备执行 1-2s（PDM 锁预检） "分配到 WS-3"
EXECUTING Worker 正在调用 SW/PDM API 5-30s（取决于任务复杂度） "执行中 60%"
REPORTING 执行完成，LLM 生成报告 2-3s（LLM API 调用） "生成报告中"
DELIVERING 推送报告到 Add-in 和飞书 1-2s "通知已发送"
DONE 全流程完成 — "完成 ✓" + 展示报告
FAILED 执行过程中发生不可恢复错误 — "失败 ✗: {错误原因}"
CANCELLED 工程师在 QUEUED 状态下主动取消 — "已取消"

5.4 错误处理策略

错误场景处理方式通知
PDM 文件被他人检出（锁冲突） 指数退避重试 (5s → 15s → 45s)，最多 3 次 重试失败 → 通知工程师："文件被 [用户名] 检出中，请稍后再试"
Worker 宕机/离线 心跳超时 (10s) → 标记 Worker OFFLINE，任务标记 FAILED 通知工程师："工作站异常，任务执行失败。已通知管理员"
SW 崩溃（Worker 侧） Worker 本地自动重启 SW（需 Worker 端实现），超时后标记 FAILED 通知工程师："SW 异常，任务执行失败"
LLM API 超时/限流 重试 1 次，仍失败则降级（跳过 LLM 编排，使用缓存计划或固定流程） 通知工程师："部分属性需要人工确认"
属性写入 PDM 后发现值错误 LLM 报告中标注"需工程师确认"项，不做自动回滚 报告中用醒目标记，工程师在 PDM 中手动修正

6. 多工作站调度与负载均衡

6.1 Worker 状态模型

状态含义可接受新任务触发转换
IDLE SW 空闲，可接受任务 是 任务完成 / 心跳恢复
BUSY 正在执行任务 否 分配新任务
OFFLINE MCP Server 不可达 否 心跳超时 (10s 无响应)
ERROR SW 崩溃或 PDM 连接断开 否 Worker 上报错误状态
MAINTENANCE 工程师手动标记维护中 否 通过 API 手动切换

6.2 心跳机制

Worker Node (每 3s):
  POST /api/v1/worker/heartbeat
  {
    "worker_id": "ws-3",
    "hostname": "HW-STATION-03",
    "status": "IDLE",          // 或 BUSY (附带当前 task_id)
    "sw_version": "2022",
    "pdm_connected": true
  }

Server:
  收到心跳 → 更新 workers 表的 last_heartbeat 时间戳
  后台线程每 10s 扫描:
    if last_heartbeat > 10s ago → 标记 OFFLINE
    if 当前有任务在此 Worker → 任务标记 FAILED + 通知工程师

6.3 调度策略

新任务进入队列
       │
       ▼
  筛选 IDLE Worker ──── 0 个 ──► 继续排队
       │
       ▼ (1 个或多个)
       │
  ┌────┴──────────────────────────────┐
  │  调度优先级 (按顺序判断):           │
  │                                    │
  │  1. 文件亲和性                     │
  │     同一装配体的零件优先分配给      │
  │     已处理过该装配体的 Worker      │
  │     (PDM 本地缓存命中率高)         │
  │                                    │
  │  2. PDM 锁预检                     │
  │     调 Worker 预检文件状态         │
  │     如果锁冲突 → 跳过该 Worker     │
  │                                    │
  │  3. Least Loaded                    │
  │     选累计完成任务数最少的          │
  │     (均匀分布负载)                 │
  │                                    │

  │     按顺序轮询                     │

关于文件亲和性 同一装配体的多个零件如果分配给同一台 Worker 处理，该 Worker 的 PDM 本地缓存已经下载了相关文件，检出速度更快。可以在 Worker 上维护一个简单的"近期处理文件列表"，调度时查询是否有命中。

7.4 热加载机制

事件行为生效时机
工程师编辑并保存 .md 文件 Server 端 watchdog 检测到文件修改，重新读取该文件内容，更新内存缓存 下一个提交的任务立即使用新规则（无需重启 Server）
新增 .md 文件到某个规则目录 watchdog 检测到文件创建，加载到缓存 立即生效
删除 .md 文件 watchdog 检测到文件删除，从缓存移除 立即生效
新增规则目录（如新增 bom/） 需在配置中注册新目录的映射关系，然后重启 Server 或调 /rules/reload 重启后生效
维护体验 工程师的工作流：打开 .md 文件 → 编辑 → 保存。不需要登录任何平台，不需要懂 YAML/JSON 格式，不需要重启任何服务。下次提交任务时 Agent 就会使用新规则。修改历史通过 git 追溯，改错了可以 revert。

8. LLM Agent 编排策略

8.1 核心设计：编排与执行分离

  传统理解 (导致单任务瓶颈):
  ─────────────────────────────
  工程师 → LLM(理解+编排+等待执行+生成报告) → Worker → 返回 → LLM(继续) → ...
  整个过程 LLM 被一个任务占用，无法处理其他任务


  本方案 (编排与执行分离):
  ─────────────────────────────
  
  阶段A: LLM 编排 (2-5s，生成执行计划)
  ──────────────────────────────
  输入: 任务上下文 + 规则文本 + 文件信息
  输出: JSON 执行计划 (工具调用列表)
  LLM 释放 → 可以处理下一个任务的编排

  阶段B: 异步执行 (5-30s，不占 LLM)
  ──────────────────────────────
  按计划顺序调用 MCP Worker
  完全 async，等待 Worker 时 CPU 释放
  多个任务的执行阶段可以完全并行

  阶段C: LLM 报告 (2-3s，生成人类可读报告)
  ──────────────────────────────
  输入: 执行结果 (改前/改后数据)
  输出: Markdown 报告文本
  LLM 释放

  流水线效果:
  ──────────────────────────────
  时间 →  0s      5s      10s     15s     20s     25s     30s

  Task-A: [编排A][    执行A(WS1)    ][报告A]
  Task-B:        [编排B][    执行B(WS2)    ][报告B]
  Task-C:              [编排C][    执行C(WS3)    ][报告C]
  Task-D:                    [编排D][    执行D(WS1)    ][报告D]

  LLM 在任何时刻只被一个阶段占用
  但不同任务的执行阶段完全并行
  实际并发 = Worker 数量 (5台 = 5个任务同时执行)

8.2 何时使用 LLM vs 固定流程

任务类型场景执行路线LLM 调用次数说明
属性填写 标准件 / 文件名前缀可匹配 Fixed Pipeline（缓存计划） 0（首次 1 次，后续缓存） 完全不需要 LLM，规则引擎直接计算
属性填写 常规零件（已知材料/分类） Fixed Pipeline（缓存计划） 0-1 类似零件已有缓存计划则直接复用
属性填写 非标零件 / 新材料 / 模糊场景 LLM Pipeline 1（编排）+ 1（报告） LLM 生成执行计划，执行后生成报告
属性检查 全部 Fixed Pipeline 0（仅报告需 1 次） 检查逻辑固定，报告可走 LLM 或模板
图纸审核 全部 LLM Pipeline 1（编排）+ 1（报告） 需要 LLM 理解图纸内容

8.3 执行计划缓存

  缓存 Key: 文件特征指纹 (文件名前缀 + SW材质 + 零件类型)
  缓存 Value: JSON 执行计划

  第一次处理 "LSH-M8x30 螺柱.SLDPRT" (材质: 碳钢):
  ─────────────────────────────────────────────
  1. 无缓存 → LLM 编排 (3s)
  2. 生成计划 → 缓存: key="LSH_碳钢_螺柱", value=[步骤列表]
  3. 执行计划 → 完成

  第二次处理 "LSH-M10x40 螺柱.SLDPRT" (材质: 碳钢):
  ─────────────────────────────────────────────
  1. 匹配缓存 key="LSH_碳钢_螺柱" → 命中!
  2. 直接使用缓存计划执行 → 0s LLM
  3. 完成

  长期效果:
  ─────────────────────────────────────────────
  处理越多 → 缓存命中率越高 → LLM 调用越少
  常规零件 2 周后预计缓存命中率 > 80%
  LLM 主要处理新出现的非标零件

8.4 LLM 并发控制

当前阶段（单 Agent 进程）

FastAPI 使用 Python asyncio，单个进程中 LLM 调用是异步的。编排阶段（2-5s）和报告阶段（2-3s）分别调用 LLM，调用期间其他任务的执行阶段不受影响。LLM API 本身有限流（通常 5-10 并发），在当前规模下单进程完全够用。

如果未来 LLM 编排请求排队明显（同时有多个非标零件需要处理），可扩展为 Agent 实例池（多个 async 协程各自持有独立 LLM 会话），不改变整体架构。

11.1 SQLite 表结构

tasks 表 — 任务队列

CREATE TABLE tasks (
    id              TEXT PRIMARY KEY,     -- task_id (UUID)
    task_type       TEXT NOT NULL,        -- property_fill / property_check / drawing_review
    status          TEXT NOT NULL,        -- queued / dispatched / executing / 
                                          -- reporting / delivering / done / failed / cancelled
    
    priority        INTEGER DEFAULT 3,    -- 1(最高) ~ 5(最低)
    filename        TEXT NOT NULL,
    pdm_file_id     TEXT,
    engineer_id     TEXT NOT NULL,
    
    payload         TEXT,                 -- JSON: 任务提交时的原始数据
    execution_plan  TEXT,                 -- JSON: LLM 生成的执行计划（可缓存）
    execution_log   TEXT,                 -- JSON: Worker 执行步骤日志
    changes         TEXT,                 -- JSON: 属性改前/改后对比
    report          TEXT,                 -- Markdown: LLM 生成的报告
    error           TEXT,                 -- 错误信息（失败时）
    
    assigned_worker TEXT,                 -- worker_id
    created_at      TEXT NOT NULL,
    dispatched_at   TEXT,
    started_at      TEXT,
    finished_at     TEXT,
    reported_at     TEXT
);

CREATE INDEX idx_tasks_status ON tasks(status);
CREATE INDEX idx_tasks_engineer ON tasks(engineer_id);
CREATE INDEX idx_tasks_priority_created ON tasks(priority, created_at);

workers 表 — 工作站状态

CREATE TABLE workers (
    id              TEXT PRIMARY KEY,     -- worker_id (如 "ws-1", "ws-2")
    hostname        TEXT,
    status          TEXT NOT NULL,        -- idle / busy / offline / error / maintenance
    current_task_id TEXT,                 -- 正在执行的任务ID
    sw_version      TEXT,
    pdm_connected   INTEGER DEFAULT 1,    -- 0/1
    
    recent_files    TEXT,                 -- JSON: 近期处理过的文件列表（用于亲和性调度）
    
    last_heartbeat  TEXT,
    created_at      TEXT NOT NULL
);

CREATE TABLE plan_cache (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    feature_key     TEXT NOT NULL,        -- 文件特征指纹（前缀+材质+类型）
    plan            TEXT NOT NULL,        -- JSON: 执行计划
    hit_count       INTEGER DEFAULT 0,    -- 命中次数
    last_used_at    TEXT,
    created_at      TEXT NOT NULL,
    
    UNIQUE(feature_key)
);

CREATE TABLE audit_log (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id         TEXT,
    worker_id       TEXT,
    engineer_id     TEXT,

CREATE INDEX idx_audit_task ON audit_log(task_id);

12. 安全性设计

维度风险措施
API Key 安全 LLM API Key 泄露 Key 仅存储在 Agent Server 端，不传递到任何客户端或 Worker。Add-in 和 Worker 均无法接触 API Key
内网通信 Add-in 与 Server 之间的请求被篡改 初期使用 HTTP（内网隔离）。如安全合规要求，可升级为 HTTPS + API Token 认证
操作审计 Agent 误操作修改属性无法追溯 全部操作写入 audit_log，记录改前/改后的值、操作时间、执行 Worker、触发工程师
权限控制 非授权人员提交任务 通过 engineer_id 做基础身份校验（Add-in 配置中绑定 Windows 登录名）。可扩展为 API Token
PDM 数据安全 Agent 错误检入导致数据覆盖 Agent 写入属性后工程师可在 PDM 中查看变更。如发现错误，可通过 PDM 版本历史回退

14. 实施路线图

阶段内容交付物预估时间验收标准
P0 FastAPI Server 骨架 + SQLite 任务表 + 手动 curl 端到端验证 可运行的 Server + 任务提交/查询 API 1-2 天 curl 提交任务 → SQLite 记录 → 查询返回
P1 Router + 单 Worker 对接（打通 MCP 调用链路） Server → MCP Worker → SW/PDM 操作 → 结果返回 2-3 天 提交任务 → Server 调 Worker → 自动填写一个零件的属性

P3 多 Worker 调度 + 心跳 + 负载均衡 Worker Pool Manager + 心跳机制 2-3 天 2 台 Worker 同时处理不同任务，互不干扰
P4 报告生成 + Add-in 面板 + 飞书推送 Add-in DLL + TaskPane UI + 飞书卡片 5-7 天 端到端：Add-in 提交 → 处理 → 报告展示 + 飞书通知

P6 Add-in 分发 + 工程师配置 + 上线部署 网络共享部署 + 配置指南 + 规则编写指南 3-5 天 3-5 名工程师试用 1 周，收集反馈迭代
预估总工期 P0-P3（Server 核心）约 2 周，P4-P6（客户端 + 上线）约 2 周，总计 3-4 周。P4 的 Add-in 开发工作量最大，如果团队有 C# SW Add-in 开发经验可以缩短。

15. 风险与应对

风险概率影响应对措施
Agent Server 单点故障（笔记本方案） 中 高 — 全组无法使用 优先评估企业服务器部署方案；笔记本方案需确保有线接入 + 防休眠 + UPS。Server 启动脚本做到 1 分钟可恢复
PDM 文件锁冲突频繁 中 中 — 部分任务延迟 重试 + 通知工程师；长期方案：与团队约定 "Agent 处理时段"，减少人工操作冲突
Worker 侧 SW 崩溃 中 中 — 单任务失败 Worker 本地守护进程自动重启 SW；Server 超时检测 + 任务重新入队
LLM API 不稳定 / 限流 低 中 — LLM 任务延迟 执行计划缓存降级（使用历史计划）；重试机制；可选多 LLM Provider 备用切换
规则维护质量下降（工程师胡乱编辑） 中 中 — 属性填写不准确 git 版本管理 + 变更通知（飞书推送规则变更记录）；定期审查规则质量
Add-in 与不同版本 SW 的兼容性 低 高 — 部分工程师无法使用 统一 SW 版本（当前 2022）；Add-in 兼容性测试
40+ 工程师并发提交导致队列积压 低 低 — 排队等待 队列位置透明展示（Add-in 显示排队位置和预估等待时间）；优先级机制保证紧急任务先处理
